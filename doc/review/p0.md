# DDL記述時の注意事項

## スキーマと命名
- テーブル参照は必ず `tapaton.<table>` を明示する（`search_path` に依存しない）。
- DDL ファイルは `db/ddl/<NN_xxx>/` 配下に置き、ファイル名は `テーブル名.sql` とする。
- マイグレーション実行順は `01.sh` の `files` 配列で管理する。

## カラム命名
- 監査系カラムは `created_at` / `updated_at` を使用する（`create_at` / `update_at` は使わない）。
- 監査ユーザーは `create_user_id` / `update_user_id` を使用する。

## NULL制約
- 必須項目は `NOT NULL` を付与する。
- `PRIMARY KEY` 構成列は必須扱いとし、列定義に `PRIMARY KEY` を埋め込まない。
- 外部キー列は `REFERENCES ... ON DELETE RESTRICT NOT NULL` 形式で記述する。

## 主キー・外部キーの書き方
- `PRIMARY KEY` はテーブル定義の最下部に `PRIMARY KEY (col1, col2)` 形式で記述する。
- 外部キーは以下の順序で統一する。
  `column_name TYPE REFERENCES target_table(target_col) ON DELETE RESTRICT NOT NULL`

## 変更時の整合性
- ER 図（`er図.dio`）と SQL のカラム名・制約を一致させる。
- 既存運用で指定された元DDLがある場合は、勝手に列構成を変更しない。
- 参照先テーブル/列が存在するか、`01.sh` の実行順で解決できるかを確認する。

## 確認
- 変更後は `bash -n db/ddl/<NN_xxx>/01.sh` でスクリプト構文を確認する。
- 必要に応じて `01.sh` を実行して連結 SQL（`init_<n>.sql`）を確認する。
